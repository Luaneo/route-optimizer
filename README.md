# Оптимизация мультимодальной логистики зерна

Прототип цифрового инструмента для оптимизации экспортной логистики зерна (500 тонн) с учётом экономических и временных ограничений.

## Математическая модель оптимизации

### Метод: полный перебор с ранжированием (Enumeration + Scoring)

Модель работает по следующему алгоритму:

1. **Вход**: точка отправки, точка назначения, дата доставки, параметры настройки
2. **Фильтрация**: отбираются все маршруты, соединяющие выбранные точки
3. **Оценка**: для каждого маршрута рассчитываются суммарная стоимость (руб/т) и срок (дней) по всем плечам
4. **Ранжирование** по трём критериям:

| Критерий | Описание | Ограничение |
|---|---|---|
| **Минимальная стоимость** | Самый дешёвый маршрут | Срок <= допустимое время + макс. отклонение |
| **Минимальный срок** | Самый быстрый маршрут | Стоимость <= макс. цена (если задана) |
| **Сбалансированный** | Оптимальное соотношение цены и срока | Нормализованный балл: `0.5 * norm(cost) + 0.5 * norm(time)` |

5. **Выход**: минимум три рекомендованных маршрута с разбивкой по плечам

### Замена модели

Модель реализована через паттерн **Strategy**. Базовый класс `BaseOptimizer` (`backend/optimizer.py`) определяет интерфейс:

```python
class BaseOptimizer(abc.ABC):
    @abc.abstractmethod
    def optimize(self, request, routes) -> OptimizationResponse:
        ...
```

Для замены модели достаточно создать новый класс, наследующий `BaseOptimizer`, и вызвать `set_optimizer(MyNewOptimizer())`.

## Режимы работы

Приложение поддерживает два режима данных:

| Режим | Когда использовать | Как включить |
|---|---|---|
| **Мок-данные** | Тестирование на телефоне (Expo) без сервера | Переключатель на главном экране, или `EXPO_PUBLIC_USE_MOCK=true` в `.env` |
| **Реальные данные** | Веб-версия на компьютере с запущенным бэкендом | Выключить переключатель, или `EXPO_PUBLIC_USE_MOCK=false` |

Переключение доступно прямо в интерфейсе приложения без перезапуска.

## Запуск

Нужно иметь установленные [Node.js](https://nodejs.org/en/download), [Python](https://www.python.org/downloads/) и [uv](https://docs.astral.sh/uv/getting-started/installation/).

### Backend

```bash
cd backend
uv run fastapi dev main.py
```

Сервер запустится на `http://localhost:8000`. Документация API: `http://localhost:8000/docs`.

### Frontend

```bash
cd frontend
npx expo start
```

- Веб: нажать `w` в терминале
- iOS/Android: отсканировать QR-код в Expo Go

### Переменные окружения (frontend/.env)

```
EXPO_PUBLIC_USE_MOCK=true          # true = мок-данные, false = реальный бэкенд
EXPO_PUBLIC_API_URL=http://localhost:8000  # URL бэкенда
```

## API-эндпоинты

| Метод | Путь | Описание |
|---|---|---|
| GET | `/api/locations` | Список всех точек |
| POST | `/api/locations` | Добавить точку |
| DELETE | `/api/locations/{id}` | Удалить точку |
| GET | `/api/routes` | Список маршрутов |
| GET | `/api/routes/{id}` | Детали маршрута |
| PATCH | `/api/routes/{id}/legs/{leg_id}` | Обновить ставку плеча |
| POST | `/api/optimize` | Запуск оптимизации |
| POST | `/api/tracking/start` | Начать отслеживание |
| POST | `/api/tracking/{id}/tick` | Симуляция: +1 день |
| GET | `/api/tracking/{id}` | Текущий статус груза |

## Данные

Исходные данные взяты из файла `Данные по маршрутам.xlsx` — 4 маршрута (A–D) из Рязани в Нава Шева (Индия) через Новороссийск или Балтийский порт, с вариантами ж/д и авто перевозки на первом плече.

## Симуляция отслеживания

- 1 секунда = 1 день реального времени
- При приближении к дедлайну (≤3 дня) — предупреждение
- При задержке >2 дней — рекомендация альтернативного маршрута
- История перемещений с временными метками
